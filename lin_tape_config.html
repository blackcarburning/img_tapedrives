🔗 Persistent Device Binding (udev)

Create stable symlinks under /dev/lin_tape/ so Storage Protect always finds the same device paths, even if kernel device numbering changes across reboots.

ℹ GB-UXB-LTSMP-P1 Implementation Note: The rules below are the actual deployed rules for the Spectra Logic TFinity Library on GB-UXB-LTSMP-P1. Key differences from a generic template: the rules file is /etc/udev/rules.d/98-lin_tape.rules, the serial number attribute is ATTRS{serial_num} (read from /sys/class/lin_tape/IBMtapeX/serial_num), symlinks are created under /dev/lin_tape/, and there is no dual-path matching (alternate_pathing=0).

Discover Serial Numbers via sysfs

How to find serial numbers: The serial_num attribute used in the udev rules comes from the lin_tape sysfs interface at /sys/class/lin_tape/IBMtapeX/serial_num. Run the command below after loading lin_tape to enumerate all drives and their serial numbers. Use these values to populate or verify your udev rules.
List all IBMtape devices with serial numbers (via sysfs)
for i in $(ls /sys/class/lin_tape/ | grep IBMtape | grep -v n); do echo "$i: $(cat /sys/class/lin_tape/$i/serial_num)"; done
Get serial number for a specific device
cat /sys/class/lin_tape/IBMtape0/serial_num
Walk sysfs attributes (confirm ATTRS{serial_num} match key)
udevadm info --attribute-walk --name=/dev/IBMtape0 | grep serial

🔀 Dual-Pathed Tape Drives

Configure IBM lin_tape alternate pathing for dual-connected tape drives. When enabled, the driver consolidates multiple physical paths into a single logical device — failover is handled transparently at the driver level, so IBM Storage Protect paths do not need to change.

⚠ GB-UXB-LTSMP-P1 Finding — alternate_pathing=1 did NOT work: On the Spectra Logic TFinity library, all drives showed Alternate: NA in /proc/scsi/IBMtape when alternate_pathing=1 was set — the lin_tape driver did not detect or consolidate alternate paths for these drives. As a result, alternate_pathing=0 is used instead. Stable device naming across reboots is guaranteed by the udev persistent symlink approach under /dev/lin_tape/ (see the Persistent Device Binding (udev) section). The content below is retained for reference for systems where alternate pathing does function correctly.

How Alternate Pathing Works

How it works: When alternate_pathing=1 is set, the lin_tape driver detects that two (or more) physical SCSI paths lead to the same tape drive (identified by serial number and WWN). Instead of creating separate device files (e.g., /dev/IBMtape0 and /dev/IBMtape1), the driver consolidates them into a single logical device (/dev/IBMtape0) with a primary and alternate path. If the primary path fails (HBA failure, switch failure, cable issue), the driver automatically fails over to the alternate path — the device file name does not change. This means IBM Storage Protect does not need any path reconfiguration — the DEFINE PATH entries continue to point to the same /dev/IBMtape0 device.

Enable Alternate Pathing

Check current alternate_pathing status
cat /sys/module/lin_tape/parameters/alternate_pathing 2>/dev/null
Create/edit modprobe config for lin_tape
cat > /etc/modprobe.d/lin_tape.conf <<'EOF'
options lin_tape alternate_pathing=1
EOF
Reload lin_tape with alternate pathing (requires stopping all tape I/O first)
systemctl stop lin_tape 2>/dev/null; rmmod lin_tape 2>/dev/null; modprobe lin_tape; systemctl start lin_tape 2>/dev/null
Verify alternate_pathing is now enabled
cat /sys/module/lin_tape/parameters/alternate_pathing
⚠️ Warning: Unloading and reloading lin_tape will temporarily remove all /dev/IBMtape* and /dev/IBMchanger* device files. Ensure all tape I/O is quiesced before reloading — stop any running SP operations, dismount tapes, and ensure no processes have tape device files open. On a running SP server, use DISABLE SESSIONS and HALT first.

Verify Primary & Alternate Paths

View all tape paths (primary/alternate)
cat /proc/scsi/IBMtape 2>/dev/null
View all changer paths (primary/alternate)
cat /proc/scsi/IBMchanger 2>/dev/null
List consolidated device files
ls -la /dev/IBMtape* /dev/IBMchanger* 2>/dev/null

for dev in /dev/IBMtape[0-9]*; do [ -e "$dev" ] || continue; echo "=== $dev ==="; udevadm info --query=property --name="$dev" 2>/dev/null | grep -E 'SERIAL|SCSI|DEVPATH'; echo ""; done
✅ Expected result: With dual-pathed drives and alternate_pathing=1, you should see fewer device files than physical SCSI paths. For example, 4 physical paths to 2 drives should show only /dev/IBMtape0 and /dev/IBMtape1. The /proc/scsi/IBMtape file will show all paths with their Primary/Alternate designation.

What Happens Without Alternate Pathing

❌ Without alternate_pathing=1: Each physical path creates a separate device file. Two paths to one drive creates /dev/IBMtape0 AND /dev/IBMtape1, both pointing to the same physical drive. This causes confusion, potential reservation conflicts, and requires careful manual management of which device file SP uses. You may also see errors like RESERVATION CONFLICT in dmesg if both paths are used simultaneously.
Check if you have duplicate serials (indicates missing alternate_pathing)
for dev in /dev/IBMtape[0-9]*; do [ -e "$dev" ] || continue; serial=$(udevadm info --query=property --name="$dev" 2>/dev/null | grep 'ID_SCSI_SERIAL' | cut -d= -f2); echo "$dev: $serial"; done | sort -t: -k2 | uniq -d -f1

SP Integration — No Path Changes Required

Key point: Because lin_tape handles dual pathing at the driver level, the device files presented to IBM Storage Protect remain unchanged. Your existing DEFINE PATH and DEFINE DRIVE commands continue to work. If a path fails, failover is automatic and transparent — SP sees no interruption (the tape operation may briefly pause and then resume on the alternate path). No UPDATE PATH or DELETE PATH / DEFINE PATH commands are needed.
Verify SP paths still point to correct device files
dsmadmc -id=ADMIN -pa=PASSWORD "QUERY PATH FORMAT=DETAILED" | grep -E "Source Name|Destination Name|Device|Online"
Check SP drive status (should remain ONLINE)
dsmadmc -id=ADMIN -pa=PASSWORD "QUERY DRIVE *"
Simulate path failure test (check dmesg after pulling one FC cable)
dmesg | grep -i -E "lin_tape|failover|alternate|path" | tail -20

🔄 Restore SP Database from Tape

Step-by-step disaster recovery procedure: restore the IBM Storage Protect server database from tape backups using dsmserv restore db. Covers preparing the volume history file, device configuration file, and the complete restore workflow. Linux only.

⚠️ Disaster Recovery Procedure: This section covers restoring the SP database — a destructive operation that replaces the current database. Only use this when recovering from a database loss, migrating to new hardware, or performing a planned DR test. The SP server must be stopped before starting a restore.

Prerequisites & Planning

You need three things to restore the SP database:
  1. Volume History file (volhist.out) — Records which tapes/volumes contain your DB backups. Must come from the OLD server.
  2. Device Configuration file (devconfig.out) — Defines the tape library, drives, and paths. On a new/replacement server, this is generated after defining library/drives/paths (see the Storage Protect Configuration section).
  3. The physical backup tapes — The actual media referenced in the volume history.

These files should be backed up regularly and stored off-server. If they are lost, you may be able to reconstruct them, but this significantly complicates recovery.

Check if volhist exists on this server
ls -la /opt/tivoli/tsm/server/bin/volhist.out 2>/dev/null; grep -i volumehistory /opt/tivoli/tsm/server/bin/dsmserv.opt 2>/dev/null
Check if devconfig exists on this server
ls -la /opt/tivoli/tsm/server/bin/devconfig.out 2>/dev/null; grep -i devconfig /opt/tivoli/tsm/server/bin/dsmserv.opt 2>/dev/null
Check dsmserv.opt for file locations
grep -iE "devconfig|volumehistory" /opt/tivoli/tsm/server/bin/dsmserv.opt 2>/dev/null

cp -v /opt/tivoli/tsm/server/bin/volhist.out /opt/tivoli/tsm/server/bin/volhist.out.bak.$(date +%Y%m%d) 2>/dev/null; cp -v /opt/tivoli/tsm/server/bin/devconfig.out /opt/tivoli/tsm/server/bin/devconfig.out.bak.$(date +%Y%m%d) 2>/dev/null

Understanding the Volume History File

The volume history file tracks every sequential volume operation — most importantly, which tapes hold your database backup volumes. Each line records a backup type (BACKUPFULL or BACKUPINCR), timestamp, volume name (tape label), device class, and series/sequence numbers. The dsmserv restore db command reads this file to know which tapes to request for mounting.
View the volume history file

Show only DB backup entries in volhist

Count full and incremental DB backups recorded
echo "Full backups: $(grep -ci BACKUPFULL /opt/tivoli/tsm/server/bin/volhist.out 2>/dev/null)"; echo "Incremental backups: $(grep -ci BACKUPINCR /opt/tivoli/tsm/server/bin/volhist.out 2>/dev/null)"
Find the most recent full DB backup entry

Always keep off-server copies of volhist.

Understanding the Device Configuration File

The device configuration file contains the DEFINE LIBRARY, DEFINE DRIVE, and DEFINE PATH statements that describe your tape hardware to the SP server. During a restore, dsmserv reads this file to know how to access the tape drives to read the backup tapes. If you're restoring on different hardware, you must edit this file to reflect the new device paths.
View the device configuration file
cat /opt/tivoli/tsm/server/bin/devconfig.out 2>/dev/null
Show library definitions
grep -i "DEFINE LIBRARY" /opt/tivoli/tsm/server/bin/devconfig.out 2>/dev/null
Show drive definitions
grep -i "DEFINE DRIVE" /opt/tivoli/tsm/server/bin/devconfig.out 2>/dev/null
Show path definitions
grep -i "DEFINE PATH" /opt/tivoli/tsm/server/bin/devconfig.out 2>/dev/null
Copy devconfig from the server being rebuilt
scp user@old-sp-server:/opt/tivoli/tsm/server/bin/devconfig.out /opt/tivoli/tsm/server/bin/devconfig.out
Editing devconfig for different hardware: If the restore target has different tape device paths (e.g., /dev/IBMtape0 is now /dev/IBMtape2, or the library changer is on a different SCSI address), you must edit the DEFINE PATH lines to match. The library and drive names should stay the same, but the DEVICE= parameter must reflect the actual Linux device files on the restore server. Use ls -la /dev/IBMtape* /dev/IBMchanger* to find the correct device names.

Restore the Database — GB-UXB-LTSMP-P1 Actual Command

GB-UXB-LTSMP-P1 restore sequence:
  1. Issue HALT to stop the SP server
  2. Run dsmserv removedb TSMDB1 as instance owner (tsminst1)
  3. Run dsmserv restore db as instance owner (tsminst1) with the parameters below

The recov= directory holds temporary recovery log data during the restore. The on= file lists the database directories (one per line). The restorekeys=yes parameter is required when database backups are encrypted.

Complete DR Restore Script

Tip: This script automates the pre-flight checks and restore sequence. Edit the variables at the top to match your environment before running. It does NOT automatically start the restore — it pauses for your confirmation.
IBM Storage Protect — Database Restore from Tape (DR script)
#!/usr/bin/env bash
# =============================================================
# IBM Storage Protect — Database Restore from Tape
# Disaster Recovery Script (Linux)
# =============================================================
set -euo pipefail

# ---- EDIT THESE VARIABLES ----
SP_INSTANCE_DIR="/opt/tivoli/tsm/server/bin"
VOLHIST_FILE="${SP_INSTANCE_DIR}/volhist.out"
DEVCONFIG_FILE="${SP_INSTANCE_DIR}/devconfig.out"
DB_DIRS=("/tsmdb001" "/tsmdb002" "/tsmdb003")
LOG_DIRS=("/tsm_activelog" "/tsm_archlog" "/tsm_archfaillog")
SP_DB_NAME="TSMDB1"
SP_USER="tsminst1"
SP_GROUP="tsmsrvrs"
# Set to specific date/time for point-in-time restore, or leave empty for latest
RESTORE_TODATE=""    # e.g., "03/01/2026"
RESTORE_TOTIME=""    # e.g., "23:59:59"
# ---- END EDIT ----

RED='\033[0;31m'; GREEN='\033[0;32m'; YELLOW='\033[1;33m'; CYAN='\033[0;36m'; NC='\033[0m'
info()    { echo -e "${CYAN}[INFO]${NC}  $*"; }
success() { echo -e "${GREEN}[OK]${NC}    $*"; }
warn()    { echo -e "${YELLOW}[WARN]${NC}  $*"; }
fail()    { echo -e "${RED}[FAIL]${NC}  $*"; }

echo "============================================="
echo " IBM Storage Protect — DB Restore from Tape"
echo " Host: $(hostname -f)"
echo " Date: $(date '+%F %T %Z')"
echo "============================================="
echo ""

# Check root
if [[ $EUID -ne 0 ]]; then fail "Must run as root."; exit 1; fi

# Check SP server is stopped
if pgrep -x dsmserv > /dev/null 2>&1; then
    fail "SP server (dsmserv) is still running!"
    warn "Stop it with: dsmadmc -id=ADMIN -pa=PASSWORD 'HALT'"
    exit 1
fi
success "SP server is stopped."

# Check volhist
if [[ -f "$VOLHIST_FILE" ]]; then
    success "Volume history file found: ${VOLHIST_FILE}"
    info "  Full backups:  $(grep -ci BACKUPFULL "$VOLHIST_FILE" 2>/dev/null || echo 0)"
    info "  Incr backups:  $(grep -ci BACKUPINCR "$VOLHIST_FILE" 2>/dev/null || echo 0)"
    info "  Last full:     $(grep -i BACKUPFULL "$VOLHIST_FILE" 2>/dev/null | tail -1)"
else
    fail "Volume history file NOT found: ${VOLHIST_FILE}"
    warn "Copy it from the old server or a backup location."
    exit 1
fi

# Check devconfig
if [[ -f "$DEVCONFIG_FILE" ]]; then
    success "Device config file found: ${DEVCONFIG_FILE}"
    info "  Libraries: $(grep -ci 'DEFINE LIBRARY' "$DEVCONFIG_FILE" 2>/dev/null || echo 0)"
    info "  Drives:    $(grep -ci 'DEFINE DRIVE' "$DEVCONFIG_FILE" 2>/dev/null || echo 0)"
    info "  Paths:     $(grep -ci 'DEFINE PATH' "$DEVCONFIG_FILE" 2>/dev/null || echo 0)"
else
    fail "Device config file NOT found: ${DEVCONFIG_FILE}"
    warn "Copy it from the old server or a backup location."
    exit 1
fi

# Check/create directories
for d in "${DB_DIRS[@]}" "${LOG_DIRS[@]}"; do
    if [[ -d "$d" ]]; then
        success "Directory exists: $d"
    else
        warn "Creating directory: $d"
        mkdir -p "$d"
        chown "${SP_USER}:${SP_GROUP}" "$d" 2>/dev/null || true
    fi
done

# Check tape devices
if ls /dev/IBMtape* > /dev/null 2>&1; then
    success "Tape devices found:"
    ls -la /dev/IBMtape* 2>/dev/null
else
    fail "No /dev/IBMtape* devices found. Ensure lin_tape is loaded and drives are visible."
    exit 1
fi

echo ""
echo "============================================="
echo " PRE-FLIGHT CHECKS COMPLETE"
echo "============================================="
echo ""

# Build restore command
RESTORE_CMD="dsmserv restore db devconfig=${DEVCONFIG_FILE} volhistory=${VOLHIST_FILE}"
if [[ -n "$RESTORE_TODATE" ]]; then
    RESTORE_CMD="${RESTORE_CMD} todate=${RESTORE_TODATE}"
    [[ -n "$RESTORE_TOTIME" ]] && RESTORE_CMD="${RESTORE_CMD} totime=${RESTORE_TOTIME}"
    info "Point-in-time restore: ${RESTORE_TODATE} ${RESTORE_TOTIME}"
else
    info "Restoring to most recent state (latest full + incrementals)."
fi

echo ""
warn "RESTORE COMMAND:"
echo "  ${RESTORE_CMD}"
echo ""
read -rp "    Proceed with restore? This will REPLACE the current database. [y/N]: " reply
if [[ "${reply,,}" != "y" && "${reply,,}" != "yes" ]]; then
    info "Restore cancelled."
    exit 0
fi

echo ""
info "Starting restore..."
cd "${SP_INSTANCE_DIR}"
eval "${RESTORE_CMD}"

echo ""
echo "============================================="
echo " Restore complete. Start the server with:"
echo "   cd ${SP_INSTANCE_DIR} && dsmserv"
echo " Then verify with:"
echo "   dsmadmc -id=ADMIN -pa=PASSWORD 'QUERY STATUS'"
echo "   dsmadmc -id=ADMIN -pa=PASSWORD 'QUERY DB F=D'"
echo "============================================="
